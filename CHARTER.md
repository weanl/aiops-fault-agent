## 0. 一句话定位（不要先讲 Agent 架构）

> **以软件运维场景为牵引，以告警、指标、日志、拓扑、知识、历史案例等数据为基础，通过智能体编排、工具调用、证据链校验和持续评测，支撑核心业务系统的故障发现、诊断分析、影响评估、处置建议和报告生成。**
>
> **主线**：场景牵引 → 用户旅程 → 数据资产 → 技术选型 → 架构设计 → 数据驱动实现 → 开发与评测闭环

**核心原则**：先定义场景和用户旅程，再反推数据、工具、模型、架构和评测。

---

## 1. 背景与问题

- 软件运维复杂度持续上升（业务系统 + 接口 + 网元 + 数据流）
- 告警 / 指标 / 日志 / 知识 / 历史案例彼此割裂
- 人工诊断强依赖专家经验（一个人离职 = 一块业务诊断能力塌陷）
- **历史案例复用不足**：每次故障都从头查，相同根因反复出现

---

## 2. 建设目标（P0 可闭环）

**P0 唯一目标**：把 3 个高频场景跑通端到端（用户旅程 → 数据查询 → 诊断 → 报告 → 反馈沉淀），形成"最小可复用闭环"。

| # | 目标 | 验收口径 |
|---|------|---------|
| G1 | 3 个 P0 场景端到端完成率 > 80% | [SCENARIOS.md §评测指标](./SCENARIOS.md) |
| G2 | 根因 Top-3 召回率 > 90% | （评测集 ≥ 50 标注样本）|
| G3 | 报告结构合规率 100% | （模板卡 8 节）|
| G4 | 评测闭环可跑通 | （评测集 + Badcase 集 + 自动评分脚本）|
| G5 | 历史案例沉淀机制 | （每个场景产生 ≥ 10 条新案例入库）|

**范围之外（P0 不做）**：
- ❌ 完全自主 ReAct Loop
- ❌ NL2SQL
- ❌ 自动执行处置（只给建议）
- ❌ 多 Agent 协作
- ❌ Mobile Money 产品（先 CBS Domain Pack）
- ❌ 受控处置（L4/L5 自主等级）

---

## 3. P0 场景范围（聚焦 CBS）

**3 个 P0 场景**（详细用户旅程参见 [SCENARIOS.md](./SCENARIOS.md)）：

| # | 场景 | 复杂度 | SRE 高频度 |
|---|------|:------:|:----------:|
| S1 | CBS 用户充值失败 | 中（单接口单实例） | ⭐⭐⭐⭐⭐ |
| S2 | CBS 交易成功率下降 | 高（多业务多接口） | ⭐⭐⭐⭐ |
| S3 | 告警风暴 / 关键告警分析 | 中（聚类 + 根告警） | ⭐⭐⭐⭐ |

**数据资产 MVP 范围**（P0 必须有，不做更多）：
- ✅ 告警（DigitalView / 内部告警平台）
- ✅ 指标 KPI（CBS 自带监控 + Prometheus）
- ✅ 错误码统计（CBS 接口日志）
- ✅ 对象拓扑（CMDB + 拓扑服务）
- ✅ 故障模式库（专家整理 30 条起步）
- ✅ 相似案例库（历史 incident ≥ 100 条）

**质量要求**：所有数据**必须可被工具访问**（不能只是"我们有数据"）

## 4. 技术方案（最小可用）

### 4.1 Agent 模式
**Recipe-driven + Tool-grounded + Evidence-based Verification**

```
固定 Recipe → 工具调用 → LLM 分析生成 → 证据链校验 → 输出报告
```

**不是 ReAct**——每场景的 Recipe 在 SCENARIOS.md §Step 5 已经写死。

### 4.2 工具集合（P0 阶段，10 个起步）

| # | 工具 | 数据资产 | 复用 |
|---|------|---------|:----:|
| T1 | `alert_query_by_time_window` | 告警 | 3 场景 |
| T2 | `kpi_trend_query` | 指标 | 3 场景 |
| T3 | `error_code_statistic` | 错误码 | 2 场景 |
| T4 | `topology_upstream_downstream` | 拓扑 | 3 场景 |
| T5 | `cmdb_object_lookup` | CMDB | 3 场景 |
| T6 | `fault_pattern_library` | 故障模式 | 3 场景 |
| T7 | `similar_incident_retrieve` | 相似案例 | 3 场景（**只生成候选**）|
| T8 | `biz_metric_overview` | 业务指标 | 场景 2 |
| T9 | `alert_burst_query` | 告警风暴 | 场景 3 |
| T10 | `recent_change_lookup` | 变更 | 场景 2 |

### 4.3 模型策略

| 任务 | 模型 |
|------|------|
| 意图识别 / 参数抽取 | 小模型（9B / M3 即可）|
| 复杂诊断分析 | 大模型（M3 main / 必要时 OpenAI）|
| 异常检测 / 告警聚合 | 规则 + 工具 |
| 知识检索 | Embedding（8765 服务，已闭环）|
| **报告生成 + 证据校验** | 大模型 + 规则（**双校验**）|

**重要**：**诊断准确性不能只依赖大模型**——必须依赖数据 / 工具 / 规则 / 校验四件套。

### 4.4 数据查询策略
- **不主打 NL2SQL**
- 短期：OpenAPI / Service API / ES 查询 / 指标服务接入（已封装为工具 T1-T10）
- 中期：语义层（**P1 再说**）

## 5. 架构设计（P0 最小可用）

**只做 4 层，不做 7 层**（避免过度工程）：

```
┌─────────────────────────────────────────┐
│ 入口层      DigitalView / DVCopilot /   │
│             告警事件触发 / OpenAPI        │
└─────────────────┬───────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│ 场景编排层  场景识别 + Recipe 路由        │
│             (从 SCENARIOS.md 自动加载)    │
└─────────────────┬───────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│ 工具执行层  T1-T10 + Evidence Store      │
│             (OpenAPI 封装 + L2 缓存)      │
└─────────────────┬───────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│ 报告输出层  模板渲染 + 校验标记           │
│             + 用户反馈采集               │
└─────────────────────────────────────────┘
```

**P0 不做**：
- 独立的"上下文构建层"——固定 Recipe 已包含
- 独立的"智能分析层"——Recipe 里写死
- 独立的"证据校验层"——嵌入到工具执行层（每工具自带校验）

**P1 才升级到 7 层**（Vanson 23:24 架构图）。

## 6. 开发路径（4 阶段，按数据闭环推进）

### 阶段 1：数据闭环验证（Week 1-2）
**目标**：**确认数据链路可用**（不是证明模型多聪明）

```
选定 1 个场景（CBS 充值失败）→ 用户旅程 → 数据查询 → 工具结果 → 报告 → 人工反馈 → 评测样例
```

**关键产出**：
- 数据查询工具 T1-T7 跑通（OpenAPI 接入 + 单元测试）
- CBS 充值失败 Recipe v1 跑通（[recipe-cbs-charge-v1.md](./recipe-cbs-charge-v1.md)）
- 评测样例 ≥ 10 条（人工标注根因 + 证据）

### 阶段 2：3 场景端到端（Week 3-4）
**目标**：3 场景并行跑通
- 场景 1（CBS 充值失败）= 主线
- 场景 2（CBS 交易成功率下降）= 业务级
- 场景 3（告警风暴）= 告警级
- 评测样例扩到 ≥ 30 条

### 阶段 3：通用 Harness 抽象（Week 5-6）
**目标**：把 3 场景中重复能力抽到通用层
- 统一报告生成器
- 统一证据校验器
- 统一反馈采集器
- 评测样例扩到 ≥ 50 条

### 阶段 4：评测闭环 + CBS Domain Pack（Week 7-8）
**目标**：评测可自动跑 + 知识沉淀
- 评测 Harness（自动评分脚本）
- Badcase 集（≥ 20 条反例）
- CBS Domain Pack v1（业务对象 + 关键指标 + 典型告警语义）
- 历史案例库 ≥ 100 条

## 7. 评测与验收（P0 必跑）

### 7.1 评测集设计

| 类别 | 数量 | 说明 |
|------|------|------|
| 标注正例 | 50 | 已知根因 + 标准证据链 + 标准分析路径 |
| Badcase 反例 | 20 | 已知误诊案例（用于回归测试）|
| 盲测集 | 10 | 隐藏根因，用于评估泛化能力 |

### 7.2 P0 必跑评测（5 层最小集）

| 层级 | 指标 | 目标值 |
|------|------|--------|
| 场景级 | 场景完成率 | >80% |
| 场景级 | 端到端耗时 | <10 min |
| 数据级 | 数据召回率 | >85% |
| 数据级 | **数据幻觉率** | <3%（红线，越低越好）|
| 诊断级 | 根因准确率 | >70%（场景1目标 75%）|
| 诊断级 | Top-3 召回率 | >85% |
| 报告级 | 证据完整率 | >95% |
| 报告级 | 报告结构合规率 | 100% |

### 7.3 验收红线（P0 不过不发版本）

- ❌ 数据幻觉率 ≥ 5%（任何 1 条编造数据都算 FAIL）
- ❌ 根因准确率 < 60%（根本没法用）
- ❌ 报告结构合规率 < 95%（模板不强制 = 报告质量不可控）

## 8. 风险与依赖

| # | 风险 | 影响 | 缓解 |
|---|------|------|------|
| R1 | **数据质量**（告警/KPI 缺失）| 场景成功率下降 | P0 先用真实数据跑，缺啥补啥 |
| R2 | **接口可用性**（OpenAPI 不稳定）| 工具调用失败 | 增加失败重试 + fallback 工具 |
| R3 | **历史案例标注质量** | 相似案例误导 | 人工审核 + 标注规范 + Badcase 集 |
| R4 | **权限与安全**（敏感接口调用）| 数据泄露 | 工具级 RBAC + 审计日志 |
| R5 | **模型稳定性**（幻觉 / 漏检）| 错误根因 | 强制证据链校验 + Recipe 约束 |
| R6 | **小模型准确率**（9B 可能不够）| 诊断质量 | 关键步骤大模型兜底 |
| R7 | **变更风险** | 部署后故障 | 灰度发布 + Canary + 完整回滚 |
