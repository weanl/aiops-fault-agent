## 3. 具体问题清单

### 问题 1：最终报告数据污染（命中 V5 幻觉红线）
最终报告第 3 节出现：
```
"error_code_statistic: 500% 全在 OCS-BJ-02"
```

该内容不来自 mock 数据，应为：
```
"5004 错误码 87% 集中在 OCS-BJ-02"
```

**判断**：典型的小模型生成阶段数据污染。即使前面 R2/R5 是对的，最终报告仍然把证据写坏了。**命中幻觉检测红线**。

### 问题 2：比例前后不一致
R2/R5 中 5005 错误码占比出现 **9.6%** 与 **5%** 两种写法，存在数值一致性问题。虽然不是主根因证据，但会影响报告可信度。

### 问题 3：自检结果不可信（Verifier 设计缺陷）
模型自检 V5 判断"不存在编造数据"，但最终报告存在明显数据污染。**当前 Verifier 不能只依赖模型自评**，必须增加确定性后置校验器。

### 问题 4：只读边界漂移
报告输出了 `kubectl get metrics ocs-bj-02-connection-pool` 命令和"重启 OCS-BJ-02 实例"建议。
- P0 是只读诊断，不能让模型直接给出看起来可执行的操作命令
- kubectl 不是已声明 mock 工具输出，也可能不是实际可用命令

**应改为**：建议人工确认后，通过受控运维工具检查 OCS-BJ-02 连接池实时使用率、最大连接数、等待队列长度和连接失败日志。

### 问题 5：高置信根因仍缺少直接证据
候选 #1 写了缺失证据：OCS-BJ-02 连接池实际监控值和阈值配置，但仍给了"置信度：高"。

**应改为**：置信度：高，但仍需通过连接池实时使用率、最大连接数、等待队列和连接失败日志完成最终确认。否则容易让报告从"高置信候选"变成"确定根因"。
